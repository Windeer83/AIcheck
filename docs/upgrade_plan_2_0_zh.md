# 升级规划 2.0：功能丰富与能力加强

> 日期：2026-07-01
> 阶段定位：在稳定可演示 MVP 基础上，增强论文事实核查系统的深度、可解释性、可管理性和部署可验证性。
> 当前约束：仍以用户上传 PDF 为主要证据来源，Markdown 导出继续作为主报告格式，外部学术源按后续阶段评估接入。

## 1. 如何判断系统是否已经更新

浏览器刷新只解决“前端页面重新加载”的问题，不能保证后端、Worker、数据库迁移和镜像都已经更新。

本次 2.0-A/2.0-B/2.0-C 的第一批升级已经落地了最小版本可见性：后端提供 `/api/version`，工作台顶部显示后端版本、构建 sha（如部署注入）和数据库迁移版本。后续部署时仍建议用镜像 tag/sha、Sealos 重启记录和健康检查一起确认。

### 1.1 本地开发环境

如果你运行的是 `npm run dev`：

- 只改了前端 React 页面：通常浏览器刷新即可看到变化，Next dev server 也会自动热更新。
- 改了前端依赖、Next 配置或构建配置：需要重启前端 dev server。
- 改了后端 API、Celery 任务、模型 provider 或数据库模型：需要重启 API/Worker。
- 新增 Alembic migration：需要执行 `alembic upgrade head`，否则数据库表结构不会更新。

### 1.2 Docker Compose 环境

如果是本地 Docker Compose：

1. 重新构建镜像：`docker compose up --build`。
2. 确认 API 启动日志里执行了 `alembic upgrade head`。
3. 浏览器刷新 Web 页面。
4. 如果页面仍旧，尝试浏览器强制刷新或清缓存。

### 1.3 Sealos 两应用部署

Sealos 上只刷新浏览器通常不够。完整更新链路是：

1. 代码推送到 GitHub。
2. GitHub Actions 构建新的 GHCR 镜像。
3. Sealos 应用拉取新镜像并重启。
4. Backend all-in-one 启动时执行 `alembic upgrade head`。
5. Web 应用重启后，浏览器刷新页面。

如果 Sealos 使用 `:main` 镜像标签，需要关注镜像拉取策略和应用重启；为了可追溯回滚，建议演示稳定后逐步使用 `sha-*` 镜像标签。

### 1.4 版本可见性

2.0 阶段已经新增：

- `/api/version`：返回后端版本号、构建 sha/时间（如果环境注入）、数据库迁移版本、运行环境。
- Web 顶部状态区显示当前后端版本和数据库版本。

后续仍建议增加：

- 前端构建版本。
- 部署文档记录“本次演示使用的镜像 tag / sha”。

这样以后就不用靠肉眼刷新判断系统是否更新。

## 2. 2.0 总体目标

2.0 的目标不是简单堆功能，而是让系统从“能演示”升级为“更像论文写作事实核查工具”。核心方向：

1. 事实核查更准：claim 抽取、证据检索、引用绑定、数值校验更可靠。
2. 过程更透明：运行日志、失败原因、阶段耗时和模型调用状态可见。
3. 结果更可管理：用户能确认、屏蔽、备注、重新核查单条 claim。
4. 报告更有用：报告能直接指导论文修改，而不是只展示 verdict。
5. 验收更稳定：中文 gold cases 覆盖主要风险类型，支持持续回归测试。
6. 部署更可验证：版本、迁移、镜像标签、健康检查和回滚路径清晰。

## 3. 功能增强路线

### 3.1 运行过程可观测性

当前已新增运行日志雏形，后续 2.0 应继续增强：

- 显示每个阶段的开始、完成和失败原因。
- 显示已处理 claim 数、总 claim 数、当前 claim 摘要。
- 显示检索到的 evidence 数量。
- 显示 LLM JSON 解析失败后的降级或重试记录。
- 增加阶段耗时：claim 抽取耗时、检索耗时、判定耗时、报告生成耗时。
- 支持下载运行日志，便于调试和复盘。

### 3.2 Claim 抽取增强

目标：减少“整句当 claim”或“拆分过细/过粗”的问题。

建议任务：

- 增强中文句子切分与引用识别。
- 对一句多事实做二次规则校验。
- 标记 claim 抽取置信度。
- 支持用户手动屏蔽、恢复、备注 claim。
- 支持单条 claim 重新核查，不必整段重跑。
- 对数值、方法、比较、因果、共识表达提高优先级。

### 3.3 Evidence 检索增强

目标：提高 Evidence Recall@5，让正确证据更容易进入判定。

建议任务：

- 对 claim 做查询改写：保留实体、数值、年份、指标、引用编号。
- 数值型 claim 做数字精确匹配加权。
- 引用型 claim 优先限定在被引用 PDF 中检索。
- 检索结果按文献、页码、相关度去重。
- 对摘要、结论、实验结果、图表标题设置不同权重。
- 在前端展示“为什么召回这些 evidence”。

### 3.4 引用绑定增强

当前阶段推荐数字引用按上传顺序绑定。2.0 可以加强为：

- 数字引用 `[1]`、`[2-4]`：继续按上传顺序绑定，并在 UI 明确显示绑定结果。
- 作者年份引用：用标题、作者、年份做 fuzzy match。
- 未匹配引用：明确列入“引用未绑定”清单。
- 用户手动修正引用到文献的映射。
- 报告中输出引用绑定表。

外部 DOI/Crossref 校验仍可作为 2.1 或 2.2，不建议在 2.0 第一阶段强行接入。

### 3.5 判定与风险增强

目标：让 verdict 更稳定，风险标签更像论文审查。

优先级建议：

1. 数值、年份、单位错误。
2. 引用错配。
3. 证据不足。
4. 过度概括。
5. 原文反驳。
6. 方法/结果混淆。
7. 主题错配。
8. 伪造文献风险。

建议任务：

- 数值型 claim 单独抽取数字、单位、指标。
- 判定 prompt 强制说明“相关不等于支持”。
- 聚合层保留“为什么不是完全支持”。
- 引入风险解释模板，让中文输出更稳定。
- 对低置信度结果提示人工复核，而不是给强结论。

### 3.6 前端工作台增强

当前 1.x 已有中文化、批量上传、文献删除/重试、原文高亮跳核查表、屏蔽结果和运行日志。2.0 建议继续增强：

- 文献管理：显示上传顺序、引用编号、解析页数、chunk 数、解析耗时。
- 引用绑定面板：显示 `[1] -> 文献标题`，允许用户调整。
- 核查表筛选：按 verdict、风险等级、复核状态、是否有证据筛选。
- 结果详情：显示 best evidence、候选 evidence、判定解释、风险说明。
- 单条 claim 操作：重新核查、屏蔽、确认、备注。
- 报告预览页：导出前可浏览 Markdown 结构。

### 3.7 报告增强

Markdown 仍作为主输出，但内容应更适合论文修改。

建议新增：

- 引用绑定表。
- 高风险优先处理清单。
- 数值风险清单。
- 证据不足清单。
- 人工复核记录。
- 每条 claim 的建议处理动作：保留、补证据、弱化、删除、重新找引用。
- 系统配置和证据来源声明。

## 4. 2.0 验收标准

建议将 2.0 验收拆成产品验收和工程验收。

### 4.1 产品验收

- 用户能看到运行日志逐条滚动更新。
- 用户能知道当前系统版本是否已更新。
- 用户能批量上传 PDF，并知道 `[1]` 对应哪篇文献。
- 用户能从原文高亮跳到对应 claim。
- 用户能按风险优先处理核查结果。
- 用户能确认、屏蔽、备注核查结果。
- Markdown 报告能指导论文修改。

### 4.2 工程验收

- 后端测试覆盖中文 gold cases。
- 前端 typecheck/build 通过。
- Alembic migration 可从空库跑到 head。
- Sealos Backend all-in-one 启动后 API/Worker 正常。
- 不提交 `.env`、上传文件、报告、数据库卷或真实密钥。

- 已完成：完善 run logs 基础闭环，前端实时滚动显示。

- 已完成：文献显示上传顺序引用编号。
- 引用绑定表。
- 手动调整引用绑定。

### Milestone 2.0-C：核查结果管理

- 已完成：按 verdict、风险等级、复核状态做表格筛选。
- 已完成：确认/屏蔽基础人工复核。
- 已完成：报告反映人工处理结果。

### Milestone 2.0-D：核查质量增强

- 数值/单位校验。
- 查询改写。
- evidence 去重和解释。
- verdict/risk 输出稳定化。

### Milestone 2.0-E：验收与部署固化

- 中文 gold cases 扩展。
- Sealos 两应用部署清单。
- 镜像 tag/sha 策略。
- 回滚流程。

## 6. 当前不建议立刻做的事

为避免 2.0 失控，以下功能建议后置：

- Word/LaTeX 插件。
- Zotero 集成。
- 大规模团队协作权限。
- 外部学术源深度接入。
- 扫描 PDF OCR 和图表多模态解析。
- DOCX/PDF 报告导出。

这些功能有价值，但会显著增加依赖、部署和验收复杂度。
